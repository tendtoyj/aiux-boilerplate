# Prompt Craft

서비스의 AI 채팅에 사용할 시스템 프롬프트(`prompts/system.md`)를 대화형으로 작성하거나 개선하는 스킬.

## UX Memory Protocol

### 시작 시
1. `ux-memory/project-context.md`가 존재하면 읽어서 프로젝트 맥락을 로드한다.
2. `prompts/system.md`가 존재하는지 확인한다.
   - **존재하면**: 내용을 읽고 현재 상태를 요약하여 사용자에게 보여준다. "지금 시스템 프롬프트가 이렇게 되어 있는데, 어떤 부분을 수정할까요?"
   - **존재하지 않으면**: 사용자에게 확인한다. "시스템 프롬프트 파일이 없네요. `prompts/system.md`에 새로 만들까요?"
3. `src/lib/tools/tool-config.ts`가 존재하면 읽어서 사용 가능한 도구 목록을 파악한다.

### 종료 시
1. 완성된 시스템 프롬프트를 `prompts/system.md`에 저장한다.
2. `ux-memory/activity-log.md`에 실행 기록을 추가한다:
   `| [날짜] | prompt-craft | [서비스명] | prompts/system.md | [신규/수정, 주요 변경사항] |`
3. `ux-memory/project-context.md`의 관련 정보를 갱신한다.

---

## General Rules

- 사용자와 대화할 때는 친근하게 "~해요", "~하세요", "~해주세요" 말투를 사용하고 짧게 말한다.
- 모든 대화는 유저가 특별히 다른 언어를 요청하지 않는 한, 한국어로 진행한다.
- 한 번에 하나의 질문만 한다. 여러 질문을 쌓지 않는다.
- 질문할 때 구체적인 예시나 선택지를 함께 제시한다. 추상적 질문은 추상적 답을 낳는다.
- 기존 시스템 프롬프트가 있는 경우, 잘 된 부분은 유지하고 부족한 부분만 보강한다.
- 최종 결과물을 저장하기 전, 사용자에게 보여주고 확인을 받는다.

## Workflow

### Phase 0: 현황 파악

스킬 시작 시 자동으로 수행한다.

1. `prompts/system.md` 존재 여부 확인
2. `src/lib/tools/tool-config.ts` (또는 도구 정의 파일) 존재 여부 확인

**기존 파일이 있는 경우:**
- 현재 시스템 프롬프트를 보여주고, 어떤 섹션이 있고 없는지 진단한다.
- 섹션별 상태를 간단히 표시:
  ```
  현재 시스템 프롬프트 상태:
  ✅ 역할 정의 — 있음
  ⚠️ 가드레일 — 없음
  ✅ 도구 설명 — 있음 (5개)
  ```
- "전체를 새로 쓸까요, 아니면 부족한 부분만 보강할까요?"로 방향을 정한다.

**기존 파일이 없는 경우:**
- "새로 만들게요. 먼저 서비스에 대해 몇 가지 여쭤볼게요!"로 Phase 1부터 시작.

---

### Phase 1: 서비스 맥락 파악

AI가 어떤 서비스에서, 무슨 역할을 하는지 이해한다.

**질문할 것:**
1. 이 AI가 들어가는 서비스가 뭔가요? (이름, 한 줄 설명)
2. AI가 구체적으로 뭘 해주나요? (추천, 분석, 안내, 대화 등)
3. 사용자가 AI한테 주로 뭘 물어보거나 요청할 것 같아요?

`ux-memory/project-context.md`에 이미 정보가 있으면 확인만 한다: "Match Table이라는 맛집 추천 서비스인 거죠?"

이 단계의 목표: **역할 섹션의 첫 문장**(정체성 정의)을 쓸 수 있을 만큼 이해하기.

---

### Phase 2: 캐릭터 설계

AI의 성격, 톤, 말투, 행동 규칙을 정의한다.

**2-1. 톤 & 성격 결정**

레퍼런스(`references/system-prompt-guide.md`의 "1-2. 성격 & 톤")를 참고하여 선택지를 제시한다.

질문 방식:
- "이 AI의 느낌을 한마디로 표현하면 뭘까요? 예를 들어 '동네 맛집 잘 아는 친구', '깐깐한 미식 평론가', '호텔 컨시어지' 같은..."
- 사용자가 감을 못 잡으면 톤 스펙트럼(격식/친밀/에너지/유머/단호함)에서 하나씩 골라보게 한다.

**2-2. 말투 디테일**

큰 톤이 정해지면 세부 사항을 확인한다:
- 이모지 쓸까요?
- 문장 길이는 짧게? 설명적으로?
- 리스트 형태? 문장 형태?

**2-3. 핵심 행동 규칙**

레퍼런스의 "1-3. 핵심 행동 규칙"을 참고하여, 서비스에 필요한 규칙을 함께 고른다.
- "답변 길이는 어느 정도가 좋을까요?"
- "정보가 부족하면 추측할까요, 질문할까요?"
- "추천할 때 이유도 같이 설명할까요?"

이 단계의 목표: **역할 섹션 전체**를 쓸 수 있을 만큼 구체화하기.

---

### Phase 3: 가드레일 설계

AI가 하면 안 되는 것, 예외 상황 대응을 정의한다.

레퍼런스(`references/system-prompt-guide.md`의 "2. 가드레일")를 참고하여 진행한다.

**3-1. 범위 제한**

- "이 AI가 절대 답하면 안 되는 질문이 있을까요?"
- "서비스와 관련 없는 질문이 오면 어떻게 할까요?"
- 서비스 특성에 맞는 out-of-scope 항목을 제안한다.

**3-2. 안전 규칙**

서비스에 필요한 안전 규칙을 제안하고 선택하게 한다:
- 개인정보 보호 규칙이 필요한지
- 정확성 보장 규칙이 필요한지 (가격, 영업시간 등 변동 정보가 있는 서비스)
- 유해 콘텐츠 방지 규칙

**3-3. 에지 케이스**

레퍼런스의 에지 케이스 표를 보여주고, 서비스에 해당되는 것을 고른다:
- "이 중에서 우리 서비스에서 실제로 일어날 것 같은 상황이 있나요?"
- 서비스 특성에 맞는 추가 에지 케이스가 있으면 함께 정의한다.

**3-4. 품질 가드레일**

- 응답 구조 규칙 (추천 시 이유 포함 등)
- 맥락 유지 규칙
- 후속 행동 규칙

이 단계의 목표: **가드레일 섹션**을 쓸 수 있을 만큼 규칙을 확정하기.

---

### Phase 4: 도구 연결

Phase 0에서 파악한 도구 정보를 기반으로, 시스템 프롬프트의 도구 섹션을 작성한다.

**4-1. 도구 목록 확인**

`src/lib/tools/tool-config.ts`를 읽어서 도구 목록을 보여준다:
- "현재 이런 도구들이 정의되어 있는데, 맞나요?"
- 도구가 없으면 이 Phase를 건너뛴다.

**4-2. 도구별 설명 작성**

각 도구에 대해:
1. 한 줄 설명
2. 언제 호출하는지 (트리거 조건)
3. 주의사항이나 기본값

레퍼런스의 "3. 사용 가능한 도구"를 참고하여, 좋은 예시 패턴으로 작성한다.

**4-3. 도구 호출 후 행동**

- "도구 결과를 받으면 AI가 어떻게 해야 할까요? 요약? 추가 질문?"
- 빈 결과나 에러 시 대응도 정의한다.

이 단계의 목표: **사용 가능한 도구 섹션**을 완성하기.

---

### Phase 5: 생성 & 검증

앞 단계에서 모은 내용을 조합하여 최종 `prompts/system.md`를 생성한다.

**5-1. 초안 생성**

다음 구조로 시스템 프롬프트를 작성한다:

```markdown
당신은 [서비스명]의 [역할]입니다.

## 역할

- [핵심 행동 1]
- [핵심 행동 2]
- ...

## 가드레일

- [범위 제한]
- [안전 규칙]
- [에지 케이스 대응]
- ...

## 사용 가능한 도구

상황에 맞게 다음 도구를 활용하세요:

- **도구명**: [설명]. [트리거 조건].
- ...

[도구 호출 후 행동 지침]
[기본값/폴백 규칙]
```

**5-2. 사용자 검토**

완성된 프롬프트를 보여주고 피드백을 받는다:
- "이렇게 완성했는데, 수정하고 싶은 부분이 있나요?"
- 수정 요청이 있으면 반영하고 다시 보여준다.

**5-3. 검증 체크리스트**

저장 전 다음을 확인한다:
- ✅ 역할이 명확하게 정의되어 있는가?
- ✅ 톤과 말투가 구체적으로 기술되어 있는가?
- ✅ 서비스 범위 밖 요청에 대한 대응이 있는가?
- ✅ 도구가 있다면, 각 도구의 용도와 호출 조건이 명시되어 있는가?
- ✅ 도구 호출 후 행동이 정의되어 있는가?
- ✅ 토큰 수가 적절한가? (너무 길면 비용 증가, 너무 짧으면 지시 부족)

**5-3. 저장**

확인을 받으면 `prompts/system.md`에 저장한다.

---

## 수정 모드

기존 시스템 프롬프트가 있고 사용자가 "부분 수정"을 선택한 경우:

- Phase 0의 진단 결과에서 부족한 섹션만 해당 Phase로 진행한다.
- 예: 가드레일만 없으면 Phase 3 → Phase 5로 바로 간다.
- 기존 내용은 최대한 유지하고, 새로 작성한 섹션만 추가/교체한다.

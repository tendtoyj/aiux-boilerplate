# Context Setup

> **Trigger keywords**: 프로젝트 시작, 컨텍스트 설정, UX 시작, 프로젝트 세팅, 맥락 정리, 프로젝트 초기화, context setup, ux setup, 새 프로젝트, 컨텍스트 업데이트

UX 프로젝트의 전체 맥락을 스캔하고 `ux-memory/`를 생성·관리하는 스킬. 이후 모든 UX 스킬이 이 메모리를 참조하여 컨텍스트를 유지한다.

이 스킬은 컨텍스트를 **수집하고 기록**하는 역할만 한다. 스킬 추천이나 라우팅은 오케스트레이터의 역할이다.

---

## General Rules

- 사용자와 대화할 때는 친근하게 "~해요", "~하세요", "~해주세요" 말투를 사용하고 간결하게 말한다.
- 모든 대화는 사용자가 다른 언어를 요청하지 않는 한 한국어로 진행한다.
- 정보를 수집할 때 한 번에 3개 이내의 질문만 한다.
- 사용자가 이미 알고 있는 정보를 반복해서 묻지 않는다.

---

## Step 1: 프로젝트 상태 스캔

프로젝트 루트에서 다음을 확인한다:

### 1-1. UX Memory 존재 여부

```
{project-root}/
├── ux-memory/
│   ├── project-context.md   → 존재 + 내용 있음?
│   └── activity-log.md      → 존재 + 기록 있음?
```

### 1-2. UX Output 존재 여부

```
{project-root}/
├── ux-output/
│   ├── {feature-name}/      → 어떤 feature 폴더들이 있는지?
│   │   └── *.md             → 각 폴더에 어떤 산출물이 있는지?
│   └── research/
│       └── {research-name}/ → 어떤 research 폴더들이 있는지?
│           └── *.md         → 각 폴더에 어떤 산출물이 있는지?
```

### 1-3. 기타 프로젝트 자료

프로젝트 루트에서 UX 관련 자료가 있는지 확인:
- 기획서, PRD, 요구사항 문서
- 디자인 파일, 와이어프레임
- 기존 리서치 결과
- 코드베이스 (이미 개발 중인 서비스인지)

### 스캔 결과에 따른 분기

- **`ux-memory/` 없음** → Step 2 (신규 세팅)으로 진행
- **`ux-memory/` 있음 + 내용 있음** → Step 3 (갱신 모드)으로 진행
- **`ux-memory/` 있음 + 내용 비어있음** → Step 2로 진행 (템플릿만 있는 상태)

---

## Step 2: 신규 프로젝트 세팅

`ux-memory/`가 없거나 비어있는 경우. 대화를 통해 프로젝트 맥락을 파악하고 메모리를 생성한다.

### 2-1. 프로젝트 기본 정보

다음 세 가지를 자연스러운 대화로 파악한다:

**질문 1 — 프로젝트가 뭔가요?**
> "어떤 서비스/제품의 UX를 다루고 있나요? 간단히 알려주세요."
>
> 예시: "배달 앱의 리오더 기능을 기획하고 있어요", "신규 SaaS 대시보드를 처음부터 설계해요"

**질문 2 — 핵심 사용자는 누구인가요?**
> "이 서비스의 주요 사용자는 어떤 사람들인가요?"
>
> 예시: "20-30대 1인가구", "B2B SaaS 관리자"

**질문 3 — 지금 어떤 단계인가요?**
> "지금 어느 단계에 있나요?"
> - 아이디어 단계 (아직 구체화 전)
> - 기획 단계 (기능 정의 중)
> - 리서치 필요 (사용자 조사 계획 중)
> - 이미 진행 중 (일부 산출물 있음)

### 2-2. 추가 맥락 수집 (선택)

Step 1 스캔에서 기존 자료가 발견된 경우, 해당 자료를 읽고 맥락을 정리한다. 사용자에게 확인:

> "프로젝트 폴더에서 [발견된 자료]를 찾았어요. 이것도 컨텍스트에 포함할까요?"

### 2-3. 메모리 생성

수집한 정보를 바탕으로:

1. `ux-memory/` 디렉토리 생성
2. `project-context.md` 작성 — 수집한 프로젝트 정보로 채움
3. `activity-log.md` 생성 — 빈 테이블로 초기화
4. 필요시 `ux-output/` 디렉토리도 함께 생성

작성된 `project-context.md`를 사용자에게 보여주고 확인을 받는다:

> "프로젝트 컨텍스트를 이렇게 정리했어요. 수정할 부분이 있나요?"

---

## Step 3: 기존 프로젝트 갱신

`ux-memory/`가 이미 있고 내용이 있는 경우. 현재 상태를 보여주고 갱신할 내용이 있는지 확인한다.

### 3-1. 현재 상태 요약

`project-context.md`와 `activity-log.md`를 읽어서 현재 상태를 요약한다:

```
## 현재 프로젝트 상태

서비스: [서비스명]
도메인: [도메인]

### Feature 진행 상황
- onboarding-flow: brainstorming ✅ → userflow ✅ → screen-plan ❌
- search-redesign: brainstorming ✅ → persona ✅

### Research 진행 상황
- onboarding-research: ur-plan ✅ → survey ✅ → interview ❌

### 최근 활동
| 날짜 | 스킬 | 대상 |
|------|------|------|
| ... | ... | ... |
```

### 3-2. 갱신 확인

> "컨텍스트에서 변경되거나 추가할 내용이 있나요? 없으면 이대로 진행할게요."

사용자가 새 정보를 주면:
1. `project-context.md`를 업데이트
2. 변경 사항을 다시 확인

---

## Step 4: 완료 안내

컨텍스트 설정이 완료되면:

> "컨텍스트가 준비되었어요. 이제 어떤 UX 스킬을 실행하든 이 맥락을 자동으로 참조해요."
>
> "다음에 뭘 하면 좋을지 알고 싶으면 `/ux-skills:ux-orchestrator`를 실행해보세요."

## project-context.md 작성 형식

```markdown
# UX Project Context

## 프로젝트 개요
- 서비스명: [서비스명]
- 도메인: [도메인/산업]
- 핵심 사용자: [사용자 설명]
- 현재 단계: [아이디어/기획/리서치/개발 등]

## 진행 중인 Feature

| Feature | 폴더 | 현재 상태 | 마지막 작업 |
|---------|------|----------|------------|
| [기능명] | ux-output/[폴더명]/ | [brainstorming/userflow/screen-plan 등] | [날짜] |

## 진행 중인 Research

| Research | 폴더 | 현재 상태 | 마지막 작업 |
|----------|------|----------|------------|
| [리서치명] | ux-output/research/[폴더명]/ | [ur-plan/survey/interview 등] | [날짜] |

## 맥락 메모

[프로젝트 배경, 팀 구조, 의사결정 맥락, 제약사항 등]
```

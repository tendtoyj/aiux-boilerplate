# UR Planner — 사용자 조사 계획 수립 스킬

> **Trigger keywords**: 사용자 조사 계획, UR 계획, 리서치 플랜, 조사 설계, 조사 계획서, user research plan, UX 리서치 계획, 설문조사 계획, 인터뷰 계획, 조사 계획서 검토, 리서치 계획 첨삭

사용자와의 대화를 통해 조사 맥락을 파악하고, 단계별로 조사 계획 보고서를 작성하거나 기존 계획서를 검토/보완하는 스킬.

---

## UX Memory Protocol

### 시작 시
1. `ux-memory/project-context.md`가 존재하면 읽어서 프로젝트 맥락을 로드한다.
2. 관련 Feature의 기존 산출물(brainstorming, userflow 등)이 있다면 `ux-output/{feature-name}/`에서 확인하여 리서치 설계에 참고한다.

### 종료 시
1. 산출물을 `ux-output/research/{research-name}/research-plan.md`에 저장한다.
   - `{research-name}`은 사용자와 대화를 통해 확정한다 (e.g., "onboarding-usability-test").
   - `ux-output/research/` 디렉토리가 없으면 생성한다.
2. `ux-memory/activity-log.md`에 실행 기록을 추가한다:
   `| [날짜] | ur-planner | [research-name] | ux-output/research/[research-name]/research-plan.md | [조사 유형] |`
3. `ux-memory/project-context.md`의 "진행 중인 Research" 테이블을 갱신한다.

---

## 핵심 규칙

- **한 번에 3개 이내의 항목**에 대해서만 질문한다
- 사용자가 다량의 정보를 한 번에 입력하더라도 **바로 계획서를 제공하지 않고** 추가 확인 절차를 거친다
  - "하나씩 차근히 진행해야 더 양질의 계획서를 작성할 수 있어요. 그래도 이대로 진행할까요?"
- 사용자가 한 번에 결과를 요청해도, **반드시 Step 3(주요 질문 정의)를 거친 후** 제공한다
- 한국어로 대화한다
- 사용자의 업무 맥락(역할, 조직, 프로젝트 배경 등)을 초반에 파악한다

## Case 분류

사용자의 맥락과 최초 대화를 통해 아래 두 가지 중 하나로 분류한다. **Case란 용어는 사용자에게 언급하지 않고** 암묵적으로 진행한다.

| 구분 | 설명 |
|------|------|
| **계획서 작성** | 조사 계획을 처음부터 수립하려는 경우 |
| **계획서 검토** | 기존 계획서를 첨삭/보완하려는 경우 |

---

## Case 1: 계획서 작성하기

대화가 끝나면 사용자는 사용자 조사 계획 보고서에 들어갈 내용을 모두 얻을 수 있어야 한다.

### Step 1 — 사용자 조사 맥락 파악하기

1. 이 조사를 통해 **얻고 싶은 결과**가 무엇인지 이끌어낸다
   - '원인' 파악인지, '비율' 파악인지, '인지' 정도 파악인지 등
2. **조사의 규모와 성격**을 파악한다
   - 캐주얼하게 팀 내에서 작게 수행하는 조사인지, 예산/인력이 투입되는 본격 조사인지
3. 조사목적, 주제, 맥락을 정리하여 제시 → **사용자 확인 후** 다음 Step으로

### Step 2 — 조사 배경과 목적 정의

1. **왜 이 조사를 하게 되었는지** 배경을 질문한다
2. 두루뭉술한 표현을 **구체적으로 파고든다**
   - "사용자들이 왜 기능을 안 쓰는지 모르겠어" → 쓰다가 중단하는 건지? 아는데 안 쓰는 건지? 아예 모르는 건지?
   - "이탈률이 높은 것 같아" → 목표치 대비 높은 건지? 이탈 기준은 무엇인지?
3. 조사 결과를 **어디에 활용할 것인지** 질문한다
4. 얻어낸 답을 **현황분석, 기회요인** 두 가지 카테고리로 정리한다
5. 어떤 종류의 조사가 필요한지도 함께 제시한다
6. 조사 배경, 목적, 조사 유형을 정리 → **사용자 확인 후** 다음 Step으로

### Step 3 — 주요 질문 정의

1. 지금까지의 대화를 바탕으로 **연구질문(Research Questions) 초안**을 마련해서 제시하고, 사용자에게 일부를 선택하도록 한다
2. 연구질문별로 어떤 방법(로그 데이터 분석, 설문조사, 인터뷰 등)으로 답을 얻을 수 있을지 **표로 제시**한다
3. A/B 테스트나 통계 분석이 필요하다면 **가설을 설정**할 수도 있다
4. 사용자의 의도에 따라 연구질문을 정교화 → **최종 질문 확인 후** 다음 Step으로

### Step 4 — 연구방법 및 계획

활용 가능한 연구방법:
- 사용자 인터뷰 / 설문조사 / 통계 분석 / 로그 데이터 분석
- 사용성 테스트 / 포커스 그룹 인터뷰 / 관찰·쉐도잉
- 코크리에이티브 워크샵 / 벤치마킹 / 유저 저니매핑

주요 조합 예시:
- 인터뷰 → 설문조사
- 설문조사 → 데이터 분석
- 사용성 테스트 (단독)
- 데이터 분석 → 인터뷰
- 워크샵 → 설문조사

**반드시 포함해야 하는 항목** (상세는 `references/method-detail.md` 참고):

| 항목 | 세부 |
|------|------|
| **참가자** | 스크리닝 기준, 목표규모, 모집방법 |
| **예산** | 필요 SW/재료비, 보상 |
| **기간** | 준비기간, 모집기간, 실행기간, 데이터 분석 |
| **세부 실행계획** | 방법론별 구체적 실행 내용 |
| **결과분석 방법** | 정량/정성 데이터 분석 방법 |

- 더 필요한 내용이 있다면 사용자와 대화로 구체화한다
- Step 3의 연구문제가 해결될 수 있는 방법인지 검토하여 제시 → **사용자 확인 후** 다음 Step으로

### Step 5 — 예상결과 및 향후계획

- 예상 결과는 `references/expected-results.md`를 참고하여 **구체적이고 손에 잡히는 예시**로 제시한다
- 예상 결과는 연구질문에 대한 답변으로서 **인사이트 중심**으로 제시한다
  - 설문조사: 어떤 결과를 얻고 싶은지에 따라 문항 구성이 달라질 수 있으므로 이를 고려한 예시 제시
  - 인터뷰: 사용자 인사이트와 추후 액션이 연결되는 형식의 예시 제시

### Step 6 — 사용자조사 계획서 제공

1. Step 1-5까지의 결과를 **요약 없이** 보고서 형식으로 통합 제공한다
   - 보고서 구조는 `references/structure-template.md` 참고
2. 초안 제공 후, **Code snippet(Markdown 형식)으로 제공할지** 물어본다
3. 사용자가 검토를 원하는 경우 → Case 2로 전환하여 진행한다

---

## Case 2: 계획서 검토하기

대화가 끝나면 사용자는 충분히 보완된 조사 계획서를 얻게 된다. **검토 중에는 비판적 관점을 충분히 유지한다.**

### Step 1 — 계획서 검토 맥락 확인하기

사용자에게 질문해서 검토가 필요한 이유와 맥락을 파악한다:
- 전반적 구성이 어떤지?
- 비문이나 오탈자를 찾아야 하는지?
- 어조나 톤을 변경해야 하는지?
- 특정 부분이 부족하다고 느끼는지?
- 참고문헌이나 사례가 필요한지?
- 타인의 시선(fresh eye)으로 검토가 필요한지?

검토 목적과 맥락 정리 → **사용자 확인 후** 다음 Step으로

### Step 2 — 계획서 보완 및 검토하기

1. 사용자의 응답에 따라 검토를 진행하여 **개선사항을 제안**한다
2. 참고문헌/사례가 필요한 경우, `references/citation-principle.md`를 따라 수정/보완 내용을 제안한다
   - 참고자료와 사례는 계획의 설득력과 깊이를 더하는 '도구'
   - 목적과 관련성을 명확히 하고, 어떻게 활용할지 구체적으로 보여주는 게 핵심
   - 간결함과 신뢰도는 기본
3. 수정/보완된 내용 확인 → **사용자 확인 후** 다음 Step으로

### Step 3 — 검토된 계획서 제공

1. Step 1의 의도를 바탕으로, Step 2에서 제시된 내용으로 보완한 **최종 수정본을 제공**한다
2. **Code snippet(Markdown 형식)으로 제공할지** 물어보고, 사용자 확인 후 필요하다면 진행한다

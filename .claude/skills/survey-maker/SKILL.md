# Survey Maker

> **Trigger keywords**: 설문 만들기, 설문 작성, 설문 문항, 설문조사 설계, survey design, questionnaire, 문항 작성, 설문 검토, 설문 제작

사용자의 리서치 계획서를 기반으로 실제로 활용 가능한 수준의 설문 문항을 체계적으로 설계하고 작성한다. 대화가 종료되면 사용자는 충분히 다듬어진 설문 문항 세트를 얻게 된다.

---

## UX Memory Protocol

### 시작 시
1. `ux-memory/project-context.md`가 존재하면 읽어서 프로젝트 맥락을 로드한다.
2. 관련 Research의 기존 산출물을 `ux-output/research/{research-name}/`에서 확인한다.
   - `research-plan.md`가 있다면 읽어서 설문 설계에 반영한다.

### 종료 시
1. 산출물을 `ux-output/research/{research-name}/survey.md`에 저장한다.
   - Research 폴더가 이미 있다면 그 안에 저장한다.
   - 없다면 사용자에게 research 이름을 확인하고 폴더를 생성한다.
2. `ux-memory/activity-log.md`에 실행 기록을 추가한다:
   `| [날짜] | survey-maker | [research-name] | ux-output/research/[research-name]/survey.md | [문항 수, 분기 유무] |`
3. `ux-memory/project-context.md`의 "진행 중인 Research" 테이블을 갱신한다.

## General Rules

- 최초 대화에서 사용자가 어떤 정보를 입력하든 반드시 추가 확인 절차를 거친다.
- 사용자가 설문 계획서를 제시해야 시작할 수 있다. 계획이 없다면 무작정 시작하지 않고 계획서 입력을 요청한다.
  - 예: "계획서가 없다면 설문을 제대로 작성하기 어려워요. 이대로 진행할까요?"
- 사용자가 이미 온전한 설문 문항을 입력했다면, 검토를 원하는지 물어보고 원할 경우 Step 4로 바로 넘어간다.
  - 예: "이미 문항을 작성하셨네요. 바로 검토를 진행할까요?"
- 반드시 Step 1~5를 순서대로 거쳐 최종 설문을 완성한다.
- 모든 대화는 사용자가 다른 언어를 요청하지 않는 한 한국어로 진행한다.
- 사용자와 대화할 때는 친근하게 "~해요", "~하세요", "~해주세요" 말투를 사용하고 간결하게 말한다.

## Step 1. 설문문항 작성을 위한 기본사항 정리

계획서를 검토한 뒤, 본격적인 설문 작성 전에 사용자에게 반드시 다음 순서대로 세 가지 사항을 물어보고 답변을 받는다.

### 1-1. 톤 설정
- `references/tone-guide.md`를 참고하여 대표적인 톤 3개를 제시하고 사용자가 선택하도록 한다.
- 톤은 설문의 질문 형식, 안내 멘트 등에 영향을 끼치므로 초반에 반드시 확정한다.

**톤 유형 요약 (toneGuide.txt 기반):**

| 톤 | 지칭 | 문체 | 예시 |
|---|---|---|---|
| 친근한 톤 | 응답자님 | 해요체 | "좋아하는 기능을 선택해주세요." |
| 정중한 톤 | 귀하, 고객님 | 합쇼체 | "해당하는 항목을 선택하여 주십시오." |
| 무난한 톤 | 사용자님, 회원님 | 해요체+합니다체 | "해당하는 항목에 응답해 주세요." |
| 학술적 톤 | 연구 참여자분 | 합니다체 | "귀하의 동의 수준을 표기하여 주시기 바랍니다." |
| 직설적 톤 | 생략/'님' | 짧고 명료 | "부서 선택.", "만족도 표시." |
| 공감적 톤 | 참여자님 | 부드러운 해요체 | "괜찮으시다면, 조금 더 자세히 말씀해주실 수 있을까요?" |

### 1-2. 분기(Branch) 설정
- 모든 참가자가 동일 문항에 답하는지, 분기가 필요한지 사용자와 상의한다.
- 설문 내용과 가설을 보고 분기가 필요할 것 같다면 적극적으로 제안한다.
- UX 설문은 질문에 따라 응답자가 다르게 분기되는 경우가 많으므로 초반에 확실히 설정해야 한다.

### 1-3. 길이 및 문항 수
- 대략적인 길이와 문항 개수에 대한 정보가 있는지 물어본다.
- 없다면 전체를 구성한 후 최종적으로 불필요한 문항을 제거하는 방식으로 진행할 것을 안내한다.

### 다음 단계 전환
- 위 세 가지 사항에 대한 답변을 모두 받았다면 사용자의 확인을 받고 Step 2로 넘어간다.

## Step 2. 전체 섹션 흐름 논의하기

사용자에게 전체적인 섹션 흐름을 **표**로 제공하여 가독성을 높인다.

### 섹션 흐름 표 구성
표에는 다음 항목이 모두 포함되어야 한다:
- 섹션 번호
- 섹션명
- 예상 문항 개수
- 주요 내용 (질문 예시)
- 대상 및 특징

### 분기 처리
- 분기가 발생하는 경우, 어떤 내용의 분기가 발생할지 함께 논의한 후 분기별 흐름을 제시한다.
- 분기 처리 규칙은 아래 "설문 분기 다루기" 섹션을 참고한다.

### Flow Chart 시각화 제안 (필수)
이 과정에서 **반드시** 다음과 같이 시각화를 제안한다:
> "지금 제시된 흐름을 flow chart로 정리해서 보시겠어요? 캔버스를 활용하시고 싶다면 요청해주세요."

- Flow chart 작성 시 `templates/outline-flowchart.jsx`를 기반으로 실제 설문에 맞게 수정한다.

### 다음 단계 전환
- 사용자의 확인을 받고 Step 3으로 넘어간다.

## Step 3. 섹션별 문항 만들기

Step 1~2를 바탕으로 섹션별로 문항을 작성한다. 아래의 설문 작성 원칙을 반드시 지킨다.

### 진행 규칙
- **한 번에 한 섹션만** 진행하고, 완성되면 확인을 받은 후 다음 섹션을 진행한다.
- 분기가 있는 경우, **한 번에 하나의 분기만** 진행하고 확인 후 다음 분기로 넘어간다.
- 하나의 분기 내에서도 한 번에 한 섹션만 진행한다.
- 이 과정이 끝나면 반드시 **표로 정리**해서 제시한다.

### 섹션 유형별 가이드

#### 시작 안내멘트 / 종료 멘트
- `references/announcement-templates.md`의 안내문 템플릿에 따라 동일한 형식으로 만든다.
- 보상 관련 부분, 문의처 등의 정보를 사용자에게 확인받아 작성한다.
- 보상이 없으면 보상 언급 없이 다음과 같이 처리한다:
  > "응답자님께서 주신 소중한 의견은 [서비스명] 서비스를 발전시키는 데 큰 도움이 됩니다."

#### 스크리닝 질문
- 분기를 만드는 스크리닝 질문은 **1개로 통합**한다.
- 서비스 인지, 사용 및 행동 특성과 관련된 질문만 포함한다.
- 인구통계 관련 질문은 포함하지 않는다.
- 인지 → 현재 행동 상태 순서로 질문한다.
- **절대** 사용자의 기억에 의존하는 질문을 하지 않는다.

좋은 예:
> "현재 '마이룸' 기능을 사용하고 계신가요?" → 한 번도 안 써봤어요 / 예전에 써봤지만 지금은 안 써요 / 지금도 사용 중이에요

나쁜 예:
> "1달 내에 OO 기능을 사용해보셨나요?" (기억 의존)

#### 서비스 사용 동기/인식/행태 섹션
- 한 번에 하나의 분기만 보여준다.
- UX 관련 문항 작성 시 `references/ux-questionnaires.md`를 참고한다.
- 척도 사용 시 `references/likert-scales.md`를 참조한다.
- 사용 동기, 원인 파악이 목적인 경우 문장/명사로 된 선택지를 효과적으로 구성한다.
- **향후 사용 의향 질문은 절대 포함하지 않는다.** (예: "만약 OO 새로운 기능이 추가된다면, 사용해볼 의향이 있으신가요?" 같은 질문은 삭제)

#### 서비스 도메인 관련 사용자 특성 섹션
- 사용자의 요청이 없더라도 **반드시** 서비스 도메인에 따라 사용자 특성을 조사할 항목을 구성한다.
- 도메인별 예시는 `references/survey-types.md`를 참고한다.

#### 인구통계학적 정보 섹션
- 어떤 정보로 구성하면 좋을지 보기를 제시하고 사용자에게 먼저 확인받은 후 진행한다.

#### 보상과 개인정보 섹션
- 보상을 위한 개인정보(이름, 연락처) 수집 문항과 개인정보 활용 동의 문항을 작성한다.
- **반드시** `references/personal-info-consent.md`에 제시된 모든 안내 문구를 포함한다.

### Flow Chart 시각화 제안 (필수)
> "지금 제시된 문항 흐름을 flow chart로 정리해서 보시겠어요? 캔버스를 활용하시고 싶다면 요청해주세요."

- `templates/outline-flowchart.jsx`를 기반으로 작성한다.

## Step 4. 설문문항 검토

### 요약 정보 제시
다음 내용을 **표로 가공**하여 제시한다:
- 총 문항 수
- 섹션별 문항 수
- 예상 소요시간
- 분기별 문항 수

### Flow Chart 시각화 제안 (필수)
> "지금 제시된 문항 흐름을 flow chart로 정리해서 보시겠어요? 캔버스를 활용하시고 싶다면 요청해주세요. 1개의 문항을 1개의 네모로 표현해서 모든 문항들의 연결과 흐름을 시각화해볼게요."

- `templates/detail-flowchart.jsx`를 기반으로 작성한다.

### 검토 프로세스
1. **길이 적절성**: 불필요한 문항이 없는지 사용자와 대화를 통해 조절한다.
2. **RQ 커버리지**: 최초 계획서의 Research Question을 해소할 수 있는 데이터가 수집되는지 검토하여 표로 제시한다.
3. **수집 데이터 및 분석 가능성**: 설문 진행 시 어떤 데이터를 수집할 수 있는지, 어떤 분석이 가능한지 예시를 들어 제시한다.

### 시뮬레이션 제안
사용자에게 설문 예상 결과 분석을 '캔버스'를 활용해 시뮬레이션할지 물어본다.
- 예시 보고서 요청 시: 파이차트, 막대그래프, 도넛차트, 히트맵 등으로 시각화
- 통계 분석을 가상으로 진행하여 그래프로 제시

### 다음 단계 전환
- 모든 검토가 완료되면 확인을 받고 Step 5로 넘어간다.

## 설문 작성 원칙

### 용어 정의

| 용어 | 정의 |
|------|------|
| **문항** | 하나의 질문 + 응답 형식(선택지 또는 주관식)으로 구성. 선택지는 척도(매우 동의~전혀 동의하지 않음) 또는 문장/명사(거리가 멀어서, 불편해서...) 형식 |
| **분기** | 특정 응답에 따라 다른 문항 세트로 이동하는 흐름 |
| **섹션** | 같은 주제의 문항으로 구성된 그룹 (예: 만족도/불만요소 섹션) |

### 응답 및 답변 구성 원칙
- 사용자가 선택한 응답 톤에 따라 질문과 답변을 구성한다.
- 해당 문항이 **필수인지 아닌지** 반드시 표시한다.
- 리커트 척도 활용 시 설문 내에서 동일한 척도를 **일관성 있게** 적용한다.
  - 나쁜 예: 1번 문항에서 "매우 동의-동의-보통-비동의-매우 비동의" → 3번 문항에서 "매우 동의함-동의함-중립-동의하지 않음-매우 동의하지 않음"
- 인구통계 정보와 보상 개인정보 동의는 **설문 가장 마지막 섹션**에 배치한다.
  - 순서: 시작 안내멘트 → 사용경험 유무 → 사용성 → 사용동기 → 사용자 특성 → 인구통계 → 개인정보 동의 → 완료 멘트

### 설문 분기(Branch) 다루기

**분기 발생 시 문항 종류:**
1. 모든 응답자에게 공통 노출 질문
2. 특정 응답 결과에 따라 분기되는 질문
3. 특정 사용자 유형/조건에서만 노출되는 질문

**분기 진행 절차:**
1. 처음 분기 발생(예: A/B) → 하나의 분기(A)에 집중해서 끝까지 완성
2. 다음 분기(B)로 돌아와서 끝까지 진행
3. A에서 다시 분기 발생(A-1, A-2) → A-1을 끝까지 완성 → A-2 진행
4. 모든 분기 완성 후 전체를 통합하여 **공통 질문**과 **분기형 질문**으로 명확히 구분 정리

### templates/ — Flow Chart 코드 템플릿

사용자가 캔버스 시각화를 요청했을 때 아래 템플릿을 기반으로 실제 설문에 맞게 수정하여 제공한다.

| 파일 | 용도 | 사용 시점 |
|------|------|-----------|
| `outline-flowchart.jsx` | **섹션 단위** 흐름 차트 — 섹션명과 분기 조건만 표시하는 심플한 버전 | Step 2~3 — 섹션 흐름 시각화 |
| `detail-flowchart.jsx` | **문항 단위** 상세 흐름 차트 — 각 문항을 노드로 표현하고, 클릭 시 질문·보기를 패널에 표시 | Step 4 — 전체 문항 검토 시각화 |

**템플릿 사용 가이드:**
- 두 파일 모두 React Flow 기반이므로 `reactflow` 패키지가 필요하다.
- `initialNodes`의 `id`, `label`, `position`을 실제 설문 구성에 맞게 수정한다.
- 분기가 있으면 edge에 `label` 속성으로 분기 조건을 명시한다.
- detail 버전은 `nodeDetails` 객체에 각 문항의 `title`, `question`, `options`를 정의한다.

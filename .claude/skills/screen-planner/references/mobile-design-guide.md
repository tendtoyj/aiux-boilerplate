# 모바일 화면 설계 가이드

Screen Planner가 모바일 플랫폼 화면을 설계할 때 참조하는 원칙과 패턴.

---

## 1. 핵심 원칙: 정보구조 ≠ 화면구성

모바일에서는 정보구조의 한 단위가 거의 항상 하나 이상의 화면으로 분리된다. 데스크탑에서 패널 하나로 처리할 수 있는 정보도, 모바일에서는 별도 화면이나 바텀시트로 풀어야 자연스럽다.

**판단 기준:**
- 한 화면 = 한 포커스. 사용자가 이 화면에서 하나의 질문에 답하거나, 하나의 행동을 완료할 수 있어야 함
- 세로 스크롤은 자연스럽지만, 스크롤 안에서 맥락이 바뀌면 화면을 나눌 신호
- 화면 수가 많아지는 것 자체는 문제가 아님 — 각 화면의 역할이 명확하면 오히려 인지 부하가 줄어듦

---

## 2. 스크린 레벨 레이아웃 패턴

개별 컴포넌트 이전에, **각 화면 전체를 어떤 구조로 잡을 것인가**를 결정하는 매크로 레벨 패턴. 모바일은 데스크탑과 달리 화면 안에서 영역을 병렬 배치하기 어려우므로, 스크린 패턴이 곧 네비게이션 구조와 직결된다.

### 스크린 패턴 (화면의 구조적 뼈대)

| 패턴 | 구조 | 적합한 경우 |
|------|------|------------|
| **리스트-디테일** | 목록 화면 → 탭 시 상세 화면으로 이동 (push) | 이메일, 설정, 상품 목록 — 항목을 골라서 깊이 보기 |
| **허브 앤 스포크** | 중앙 허브에서 각 기능/섹션으로 진입, 항상 허브로 복귀 | 홈 화면, 런처 — 독립적인 기능이 여러 개인 앱 |
| **탭 기반** | 하단 탭 바로 3-5개 독립 섹션 전환. 각 탭이 자체 스택 보유 | 대부분의 소비자 앱 — 핵심 기능이 동등한 중요도 |
| **피드** | 세로 스크롤 무한 목록. 카드 단위로 콘텐츠 표시 | SNS, 뉴스, 콘텐츠 앱 — 탐색/발견 중심 |
| **풀스크린 몰입** | 콘텐츠가 화면 전체를 차지. 네비게이션은 숨김 (탭하면 등장) | 사진/영상 뷰어, 리더, 게임 — 콘텐츠에 집중 |
| **스텝 (위자드)** | 순차적 단계. 프로그레스 바로 전체 진행 표시 | 온보딩, 회원가입, 체크아웃 — 복잡한 입력을 단계별 분해 |
| **대시보드 (모바일)** | 핵심 지표 3-5개를 카드로 세로 배치. 각 카드 탭 시 상세로 | 모니터링, 건강 앱 — 한눈에 상태 확인 |
| **캔버스/지도** | 핵심 객체(지도, 캘린더)가 화면 전체. 하단이나 오버레이로 조작 | 지도 기반 서비스, 캘린더 — 공간/시간 모델과 직결 |
| **검색 중심** | 검색 바가 최상단, 결과가 아래로 나열 | 탐색 엔진, 사전, 이커머스 — "찾기"가 주 행동 |
| **Q&A / 가이드** | 질문에 답하면 다음 화면. 최종적으로 결과/추천 도출 | 추천, 진단, 맞춤형 제안 — 사용자 입력 기반 경험 |

### 화면 내 콘텐츠 배치 패턴

| 패턴 | 구조 | 적합한 경우 |
|------|------|------------|
| **싱글 컬럼 스크롤** | 모든 콘텐츠가 하나의 열에 세로로 나열 | 모바일의 기본. 대부분의 화면에 적합 |
| **카드 스택** | 카드가 세로로 쌓임. 각 카드는 독립된 정보 단위 | 피드, 대시보드, 목록 — 동등한 항목의 나열 |
| **히어로 + 콘텐츠** | 상단에 대형 이미지/비주얼 + 하단에 세부 정보 | 상품 상세, 프로필, 결과 화면 — 임팩트 있는 첫인상 |
| **그룹핑** | 관련 항목을 섹션으로 묶고 섹션 헤더로 구분 | 설정, 폼, 카테고리별 목록 — 논리적 분류가 필요한 정보 |
| **가로 스크롤 캐러셀** | 가로로 스와이프하는 카드/이미지 열 | 추천 콘텐츠, 카테고리 탐색 — 세로 공간 절약 + 발견 유도 |
| **오버레이 레이어** | 기존 화면 위에 바텀시트/모달이 겹침 | 필터, 옵션, 빠른 액션 — 맥락 유지하며 부가 작업 |

## 3. 화면 전환 패턴 — 언제 무엇을 쓰나

### 새 화면 (Push Navigation)
- 맥락이 완전히 바뀔 때 (목록 → 상세, 설정 → 하위 설정)
- 사용자가 이전 화면을 참조할 필요 없이 새로운 작업에 집중할 때
- 뒤로가기로 돌아올 수 있는 선형적 흐름
- iOS: Navigation Stack, Android: Fragment/Activity 전환

### 바텀시트 (Bottom Sheet)
- 현재 맥락을 유지하면서 추가 정보나 옵션을 보여줄 때
- 작업이 짧고 간단할 때 (필터 선택, 공유 옵션, 간단한 설정)
- 모달 바텀시트: 반드시 상호작용 후 닫아야 함 / 비모달: 배경과 동시 사용 가능
- NNGroup: "바텀시트는 모바일에서 맥락 유지 + 부가 작업의 최적 패턴"

### 인라인 확장 (Accordion / Expandable)
- 같은 화면 안에서 추가 디테일을 열고 닫을 때
- 반복적으로 여러 항목을 탐색할 때 (FAQ, 설정 그룹, 카테고리별 항목)
- 화면 이동 없이 맥락을 유지하는 것이 중요할 때

### 모달 (Full-screen Modal)
- 독립적인 작업 흐름을 시작할 때 (새 항목 생성, 편집 모드)
- 완료/취소가 명확한 작업
- 뒤로가기가 아닌 "X" 또는 "완료"로 닫힘 — 기존 네비게이션 스택과 분리

### 탭 (Tab Bar)
- 앱의 최상위 섹션 간 이동 (3-5개)
- 각 탭은 독립적인 네비게이션 스택을 가짐
- 엄지 손가락 닿는 하단에 배치

## 4. 제품 유형별 모바일 패턴

### 대시보드 / 분석형
모바일 대시보드 사용자는 **모니터링 모드** — 상태 확인, 이상 감지가 목적.

- 한 화면에 3-5개 핵심 지표만 표시 (데스크탑의 15-20개와 대조적)
- 각 화면이 정확히 하나의 질문에 답하도록 설계
- 차트는 세로 스크롤로 쌓되, 가로 스와이프로 시간 범위 조절
- 필터와 날짜 선택기는 화면 하단 40% (엄지 도달 영역)에 배치
- 상세 드릴다운은 새 화면으로 분리 — 차트 탭 시 상세 뷰로 이동

### B2C 소비자 앱
사용자는 짧은 세션(평균 2-3분), 이탈이 빠르고, 감정적 연결이 중요.

- **온보딩**: Just-in-Time 원칙 — 기능을 쓸 때 설명, 미리 다 보여주지 않음. 가치(aha moment)까지의 시간을 최소화
- **화면당 선택지 3-5개 이하**: 선택 과부하가 이탈의 가장 큰 원인
- **하단 네비게이션 바 필수**: 핵심 기능은 항상 접근 가능해야 함. 햄버거 메뉴는 발견성 31% 감소
- **진행감 표시**: 프로그레스 바, 마이크로 리워드 등으로 "진행하고 있다"는 느낌
- **개인화**: 사용자가 자기 맥락에 맞는 콘텐츠를 보고 있다는 느낌이 참여를 유지시킴

### 생산성 / 도구형
작업 완료가 목적. 효율과 실수 방지가 핵심.

- **멀티스텝 폼**: 한 화면에 모든 입력을 넣지 않음. 단계별로 나누되, 진행 표시기로 전체 그림을 보여줌
- **인라인 편집 우선**: 반복적 수정 작업은 새 화면보다 인라인이 효율적
- **실행 취소 가능**: 모바일은 실수가 잦으므로, 되돌리기를 항상 제공
- **저장 상태 명시**: 자동 저장이든 수동이든, 현재 상태를 항상 알 수 있어야 함

### 콘텐츠 / 미디어형
탐색과 소비가 주 행동. 몰입과 발견의 균형.

- **피드 패턴**: 세로 스크롤 무한 피드가 기본. 카드 단위로 콘텐츠 표시
- **상세 전환**: 카드 탭 → 전체 화면 상세. 스와이프로 다음/이전 항목 이동
- **몰입 모드**: 콘텐츠 소비 중에는 네비게이션을 숨기고, 탭하면 다시 나타남
- **저장/북마크**: 언제든 가볍게 저장할 수 있는 진입점. 나중에 볼 수 있다는 안심감

### 이커머스
모바일은 탐색/발견, 구매 전환은 데스크탑 대비 낮음. 탐색 경험을 최적화.

- **상품 이미지**: 4-6장이 최적 (1-3장 대비 전환율 23% 높음, 7장 이상은 수확 체감)
- **상품 목록**: 세로 스크롤 + 필터는 상단 고정 또는 바텀시트
- **체크아웃**: 단순 거래는 단일 페이지가 21% 높은 전환. 복잡한 배송 옵션이 있으면 멀티스텝
- **탭 타겟**: 최소 44×44px. 단일 컬럼 레이아웃
- **장바구니 접근**: 항상 접근 가능한 위치에 배지와 함께 표시

## 5. 인터랙션 특성

### 제스처
- **스와이프**: 삭제, 보관, 다음/이전 항목. 방향에 의미를 부여하되, 항상 대체 UI(버튼)도 제공
- **풀투리프레시**: 목록/피드 갱신의 표준 패턴
- **롱프레스**: 컨텍스트 메뉴, 멀티셀렉트 모드 진입. 발견성이 낮으므로 핵심 기능에 쓰지 않음
- **핀치 줌**: 이미지, 지도 등 콘텐츠 확대. 네비게이션에는 사용하지 않음

### 엄지 영역 (Thumb Zone)
- 화면 하단 1/3이 가장 쉬운 도달 영역
- 주요 액션 버튼, 네비게이션은 하단 배치
- 상단은 타이틀, 상태 표시 등 읽기 전용 정보

### 키보드 대응
- 입력 필드 포커스 시 키보드가 화면의 40-50%를 가림
- 중요한 액션 버튼이 키보드에 가려지지 않도록 설계
- 입력 완료 후 키보드 dismiss가 자연스러워야 함

## 6. 네비게이션 구조 — iOS/Android 공통 원칙

Apple HIG의 3가지 네비게이션 스타일:

| 스타일 | 특징 | 적합한 경우 |
|--------|------|------------|
| **계층형 (Hierarchical)** | 화면마다 하나의 선택 → 깊이 이동. 뒤로가기로 복귀 | 설정, 메일, 파일 탐색 등 트리 구조 |
| **평면형 (Flat)** | 최상위 카테고리 간 자유 전환 (탭 바) | 음악, 앱스토어 등 독립적 섹션이 있는 앱 |
| **콘텐츠 주도형 (Content-driven)** | 콘텐츠 자체가 네비게이션을 정의 | 게임, 인터랙티브 스토리, 지도 기반 앱 |

대부분의 앱은 이 중 2가지를 조합한다 (예: 탭 바(평면) + 각 탭 내 스택(계층)).

**핵심**: 네비게이션은 사용자가 의식하지 않을 때 가장 잘 작동한다. 구조와 목적에 맞으면 자연스럽게 느껴짐.

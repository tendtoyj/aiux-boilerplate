## Flow 1. "내가 만든 거 올려야지" — 레시피 공유

### 진입 동기

이 플로우에 진입하는 사용자는 이미 요리를 해본 상태. "올려야겠다"는 생각이 드는 순간은 크게 두 갈래:

**A. 요리 직후, 결과물이 눈앞에 있을 때**
- 만들어놓고 보니 잘 나와서 자랑하고 싶은 경우
- 누군가에게 레시피를 알려달라는 요청을 이미 받은 경우
- 이 두 가지는 공통적으로 "지금 이 순간, 바로 올리고 싶다"는 즉시성이 핵심.

**B. 나중에 정리하면서**
- 여러 번 만들면서 나만의 비율이나 팁이 쌓여서, 한번 제대로 정리해두고 싶은 경우
- 블로그나 SNS에 이미 올린 레시피를 이 앱에도 아카이빙하고 싶은 경우
- 이쪽은 "지금 당장"이 아니라 "시간 내서 제대로" 하려는 동기.

A와 B는 작성 과정에서 기대하는 경험이 다름. A는 빠르고 가벼운 기록, B는 꼼꼼하고 완성도 높은 정리. 하나의 플로우가 두 동기를 모두 수용할 수 있어야 함.

### 진입 메뉴 구조

이 플로우와 관련된 앱의 주요 진입점은 세 가지:

- **홈 화면 FAB(+) 버튼** — 어디서든 접근 가능한 글로벌 작성 진입점
- **내 레시피 탭 → 새 레시피** — 내 레시피 목록을 보다가 추가하는 맥락적 진입
- **요리 따라하기 완료 화면 → "내 버전으로 저장"** — 다른 사람 레시피를 따라 만든 후, 내가 바꾼 부분을 별도 레시피로 분리하는 경우

세 진입점 모두 같은 작성 화면으로 연결되지만, 세 번째 경우는 원본 레시피의 정보가 일부 프리필(pre-fill)된 상태로 시작됨. 어디서 왔는지에 따라 초기 상태가 달라지는 것이 자연스러움.

### 시나리오 A: 요리 직후 바로 올리기

**구체적 예시: 주말에 파스타를 만들고 사진이 잘 나와서 바로 올리는 경우**

1. **작성 시작** — 홈 화면에서 FAB 버튼을 누름. 작성 화면이 열리는데, 빈 양식이 아니라 "사진부터 올려볼까요?"라는 안내와 함께 카메라/갤러리 선택이 먼저 나옴. 요리 직후의 사용자는 사진이 가장 먼저 하고 싶은 행동이기 때문.
2. **사진 업로드** — 방금 찍은 사진 2-3장을 선택. 업로드되면 사진이 상단에 크게 보이고, 그 아래로 레시피 정보 입력 영역이 펼쳐짐. 사진이 올라간 순간 "벌써 반은 했네"라는 느낌을 줘야 함 — 텍스트 입력의 부담을 사진이 덜어주는 구조.
3. **기본 정보 입력** — 요리 이름, 간단한 소개 한 줄. 여기서 중요한 것은 카테고리나 태그 같은 분류를 이 시점에 강요하지 않는 것. "파스타"라고 이름만 쓰면 앱이 자동으로 "면 요리" 카테고리를 제안하고, 사용자는 수락하거나 무시할 수 있음.
4. **레시피 본문** — 여기서 분기:
   - **간단 모드** — 자유 텍스트로 한 덩어리에 쭉 적기. "올리브오일 넉넉히 두르고, 마늘 3쪽 편썰어서 볶다가..." 식으로 대화체로 쓸 수 있음. 앱이 나중에 구조화를 제안할 수 있지만, 지금 강요하지 않음.
   - **구조 모드** — 재료 목록 / 조리 단계를 분리해서 입력. 각 단계에 사진을 추가할 수 있음. "시간 내서 제대로"하려는 B 동기의 사용자에게 적합.
   - 기본값은 간단 모드. 사용자가 원하면 구조 모드로 전환 가능. 핵심은 A 동기 사용자가 구조 모드의 존재 때문에 부담을 느끼지 않는 것.
5. **공개 전 확인** — 작성이 끝나면 미리보기 화면. 다른 사람에게 어떻게 보일지를 보여줌. 여기서 "아, 사진 순서 바꿔야겠다" 같은 마지막 조정이 일어남. 공개 범위 설정(전체 공개 / 링크 공유만 / 비공개 저장)도 이 시점에.
6. **공개** — 올리기 버튼을 누르면 완료. 완료 화면에서:
   - 공유 링크 복사 버튼 (카카오톡, 인스타 스토리 등으로 바로 공유)
   - "레시피를 알려달라고 한 사람이 있나요?" — 특정 사람에게 직접 보낼 수 있는 옵션
   - 내 레시피 목록으로 돌아가기

**참고:**
- "간단 모드 → 구조 모드" 전환은 가능하지만 반대는 주의가 필요. 구조화된 데이터를 자유 텍스트로 합치면 정보 손실이 생길 수 있으므로, 전환 시 확인이 필요하거나 아예 비가역적으로 설계할 수도 있음.
- "내 버전으로 저장"으로 진입한 경우, 원본 레시피의 재료/단계가 프리필되어 있고 사용자가 바꾼 부분만 수정하면 됨. 원본 레시피와의 연결(출처)은 자동으로 남아, 나중에 "이 레시피는 OO님의 레시피를 참고했어요"라고 표시됨.
- 레시피 공개 후에도 언제든 수정 가능. 수정 이력이 남아서 "아, 소금 양을 줄였구나" 같은 변경 사항을 다른 사용자도 볼 수 있으면 신뢰도에 도움이 됨. 다만 이건 이 플로우의 핵심은 아니고, 관리 플로우에서 다룰 수 있는 영역.

### 핵심 가치

- 요리 직후의 "지금 바로 올리고 싶다"는 모멘텀을 놓치지 않는 것이 이 플로우의 핵심. 사진부터 시작하고 → 최소한의 텍스트만 요구하고 → 분류나 구조화는 나중에 할 수 있게 함으로써 "3분 안에 올리기"가 가능한 경험.
- 동시에, 나중에 제대로 정리하고 싶은 사용자를 위한 구조 모드가 같은 화면 안에 공존. 두 동기를 하나의 플로우가 자연스럽게 수용함.
